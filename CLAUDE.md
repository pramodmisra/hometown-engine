> Project context for Claude Code. Read this fully before making changes.

## 1. Mission

Build a web application that identifies Team USA **hubs** — geographic regions producing Olympic and Paralympic athletes — and uses Gemini to explain the conditions that could help foster athletic excellence in those regions.

This is a submission for the **Team USA × Google Cloud "Vibe Code for Gold" Hackathon**, Challenge 2: The Hometown Success Engine.

**Submission deadline:** May 11, 2026, 5:00 PM Pacific Time.
**Total prize pool:** $75,000. Challenge 2 has a dedicated $8,000 winner slot plus eligibility for the $15,000 Grand Prize and $5,000 Honorable Mentions.

### Core value proposition
A fan can enter any US zip code, state, or region and discover:
- How many Team USA Olympians AND Paralympians have come from that area
- Which sports cluster in that region
- *Why* that region might be a hub (climate, infrastructure, school programs, geography)
- How their hometown compares to other hubs nationally

The "why" is generated by Gemini synthesizing multiple public data sources. This is the AI value-add.

## 2. Hackathon Constraints — READ BEFORE EVERY COMMIT

These are non-negotiable rules from the official hackathon rules. **Violations result in immediate disqualification.** Treat this section as a compliance checklist.

### 2.1 Data restrictions
- ✅ **Permitted:** Finish placement (1st, 2nd, 3rd), medals, athlete counts by region, sport categories, hometown geography.
- ❌ **Prohibited:** Finish times, specific scoring results, performance metrics beyond placement.
- ✅ **Permitted data sources:** Official Team USA website (teamusa.com), open-source repositories filtered for Team USA only, public weather data (NOAA), public geographic data (Census, USGS).
- ❌ **Scope restriction:** Only Team USA (Olympic AND Paralympic) data. International data sets are prohibited unless filtered to US-only.

### 2.2 NIL (Name, Image, Likeness) restrictions
- ❌ **Never** display individual athlete names, photos, or videos in the UI output.
- ❌ **Never** use Team USA multimedia featuring identifiable athletes.
- ✅ Aggregate analysis is allowed. The app can *analyze* data tied to athlete names internally, but the **output must be at the aggregate level only**. Example: "23 athletes from this region across 7 sports" — never a list of names.

### 2.3 Branding restrictions
- ❌ No IOC intellectual property: no Olympic rings, no Olympic torch imagery, no Paralympic Agitos.
- ❌ No corporate branding other than Google Cloud in UI, mockups, or demo materials.
- ❌ Avoid "Olympic Games" as the app title.

### 2.4 Terminology restrictions
- "Olympic Games [City] [Year]" or "Paralympic Games [City] [Year]" — full names only.
- "LA28 Games" or "LA28 Olympic and Paralympic Games" for the upcoming event.
- Use **"Olympian"** or **"Paralympian"** — never "former" or "past" Olympian.
- Use official sport names (e.g., "Swimming") not National Governing Body names (e.g., "USA Swimming") unless explicitly authorized.

### 2.5 Generative media restrictions
- ❌ No AI-generated images of real people or athletes.
- ✅ Animations only (data visualizations, abstract animated illustrations, motion graphics).
- ❌ No Games footage or unlicensed music in the demo video.

### 2.6 AI tooling restrictions
- ✅ Only Google Cloud GenAI tools (Gemini via Vertex AI, Cloud Functions, etc.) for processing data.
- ❌ No ChatGPT, Claude API, or other non-Google GenAI for data processing in the production app.
  - Note: Using Claude Code as a developer tool to *write* the codebase is fine. The constraint is that the deployed app must use Gemini for AI features.

### 2.7 Conditional phrasing requirement
- Use phrases like **"could help find"**, **"could lead to"**, **"associated with"**.
- Avoid: "produces winners", "guarantees", "predicts performance".
- Geography correlates with athletic representation — it does not cause athletic outcomes. Every Gemini-generated narrative must use conditional language.

### 2.8 Submission requirements
- Public GitHub repo licensed under **Apache License 2.0** (visible in About section).
- Hosted, working URL deployed on Google Cloud (Cloud Run, App Engine, or AI Studio).
- Demo video ≤ 3 minutes, unlisted on YouTube, in English (or with English subtitles).
- Demo must show live product, the Google Cloud console / AI Studio / code, and how Gemini is used.
- **Do not share publicly** before/during/after — no LinkedIn, X, or YouTube posts beyond the unlisted submission video.

## 3. Differentiation Strategy

The judging rubric is Impact (40%) + Technical Depth (30%) + Presentation (30%). Every architectural decision should optimize for one of these.

### 3.1 Impact (40%) — Paralympic parity is non-negotiable
- Every hub card shows Olympic AND Paralympic athlete counts with equal visual weight.
- The featured "Hub of the Day" rotation must include Para-led hubs at minimum 50% of the time.
- The conversational agent gives equal analytical depth to Para sport classifications and infrastructure.
- Featured demo regions explicitly include Para-focused hubs (e.g., regions with strong adaptive sports infrastructure, university programs).

### 3.2 Technical Depth (30%) — Gemini doing real work
Gemini is used for **four** distinct capabilities, not just one:
1. **Narrative generation** — multi-source synthesis explaining why a region might be a hub.
2. **Structured reasoning** — correlating climate, infrastructure, demographics into ranked factors.
3. **Agentic Q&A** — conversational interface answering open-ended hometown questions with grounded citations.
4. **Discovery** — proactively surfacing surprising hub patterns (small towns with disproportionate athlete representation in specific sports).

### 3.3 Presentation (30%) — One unforgettable demo moment
The 3-minute video opens with a personalization hook: a judge enters their own hometown and watches the app generate a complete picture in real time. Maps + zip code entry + animated narrative reveal = shareable.

```
│  ├── /api/agent/ask               — conversational agent      │
│  └── /api/hometown/{zipcode}      — personalization endpoint  │
└──────────┬──────────────────────────────┬───────────────────┘
           │                              │
           ▼                              ▼
┌──────────────────────┐    ┌──────────────────────────────┐
│  Vertex AI           │    │  BigQuery                    │
│  - Gemini 2.5 Pro    │    │  - athletes_aggregate        │
│  - Gemini 2.5 Flash  │    │  - regions                   │
│  - Vector Search     │    │  - climate (NOAA)            │
│  - Embeddings        │    │  - infrastructure            │
└──────────────────────┘    │  - school_sports_programs    │
                            └──────────────────────────────┘
```

### Request flow example: "Tell me about Eugene, Oregon"
1. Frontend resolves "Eugene, Oregon" → region_id via /api/regions/lookup
2. Backend queries BigQuery for aggregate athlete counts, climate, infrastructure
3. Backend assembles a structured context blob
4. Backend calls Gemini 2.5 Pro with the context + a system prompt requiring conditional phrasing
5. Backend caches the narrative (Redis, 24h TTL)
6. Frontend renders narrative + animated map + sport breakdown chart

## 6. Data Sources

All sources must be filtered to **Team USA athletes only** (US-scope).

| Source | URL | Use |
|---|---|---|
| Team USA Official | teamusa.com | Athlete profiles (hometowns), team rosters |
| Olympedia (US-filtered) | olympedia.org | Historical participation 1904+ |
| Wikipedia (Team USA lists) | en.wikipedia.org | Cross-reference, Para athlete data |
| NOAA Climate Data | ncei.noaa.gov | County-level climate (temp ranges, precipitation, sunlight) |
| US Census | census.gov | Demographics, geographic boundaries |
| USGS National Map | usgs.gov | Elevation, terrain |
| NFHS Sports Participation | nfhs.org | Public high school sports program data |
| College Sports Associations | ncaa.org public data | Collegiate pipeline indicators |

### Data scraping rules
- Only scrape publicly available pages.
- Respect robots.txt.
- Rate-limit requests (no more than 1 req/sec to any single domain).
- Cache aggressively — re-scraping during development is wasteful.
- Store raw scraped data separately from aggregated/processed data.

### Hometown ambiguity
- Some athletes list multiple hometowns (born/raised/training base). Decision: use the **listed primary hometown** from teamusa.com, fall back to birthplace.
- Document this choice in the UI: "Hometown defined as primary listed hometown on Team USA profile."

## 7. Data Model (BigQuery)

```sql
-- Aggregate-only. No individual athlete records exposed via API.
CREATE TABLE athletes_aggregate (
  region_id STRING,           -- FIPS code or custom region ID
  region_name STRING,         -- "Eugene, OR" or "Marin County, CA"
  region_type STRING,         -- 'city', 'county', 'metro', 'state'
  sport STRING,               -- "Swimming", "Para Swimming", etc.
  is_paralympic BOOLEAN,
  athlete_count INT64,        -- aggregate count, never individual records
  era STRING,                 -- '1900-1950', '1950-2000', '2000+', 'recent'
  decade INT64,
  data_source STRING,
  last_updated TIMESTAMP
);

CREATE TABLE regions (
  region_id STRING,
  region_name STRING,
  region_type STRING,
  state STRING,
  latitude FLOAT64,
  longitude FLOAT64,
  population INT64,
  area_sq_km FLOAT64,
  elevation_m FLOAT64
);

CREATE TABLE climate (
  region_id STRING,
  avg_temp_c FLOAT64,
  temp_range_c FLOAT64,
  annual_precip_mm FLOAT64,
  sunlight_hours_yr FLOAT64,
  climate_zone STRING        -- 'temperate', 'subtropical', 'mountain', 'coastal', etc.
);

CREATE TABLE infrastructure (
  region_id STRING,
  infrastructure_type STRING, -- 'pool', 'track', 'rink', 'climbing_gym', 'adaptive_facility'
  count INT64,
  per_capita_rank INT64,
  is_adaptive BOOLEAN          -- specifically for adaptive/Paralympic facilities
);

CREATE TABLE school_sports_programs (
  region_id STRING,
  sport STRING,
  participation_count INT64,
  has_adaptive_program BOOLEAN
);

-- Vector embeddings for "similar hubs" feature
CREATE TABLE region_embeddings (
  region_id STRING,
  embedding ARRAY<FLOAT64>,    -- generated via Vertex AI text-embedding-004
  embedding_text STRING        -- the source text used to generate
);
```

### Critical: aggregate-only outputs
The API must never return individual athlete names in responses to the frontend, even if they exist in some upstream data source. All queries must `GROUP BY region + sport` and return counts.

## 9. Gemini Integration Details

### 9.1 Model selection
- **Gemini 2.5 Pro**: complex reasoning (Why-Engine, Discovery Mode, agent answers).
- **Gemini 2.5 Flash**: cheap narrative generation, summaries, follow-up questions.
- Default to Flash; escalate to Pro only when reasoning quality matters.

### 9.2 System prompt template (Why-Engine)
```
You are an analyst exploring patterns in publicly available Team USA athlete data
and US geographic/climate data. You are not making predictions about individuals
or guaranteeing performance outcomes.

CONSTRAINTS:
- Always use conditional phrasing: "could help foster", "may be associated with",
  "could contribute to". Never say "produces", "guarantees", or "predicts".
- Treat Olympic and Paralympic athletes with equal analytical depth.
- Never mention individual athlete names, even if present in the data.
- Reference data sources by category, not by URL.
- Output 3-4 short paragraphs. Use plain language.

INPUT DATA:
{structured_context_blob}

TASK:
Explain what factors in this region's environment could be associated with the
sports it has produced Team USA athletes in. Cover climate, infrastructure,
and any notable adaptive sports presence with equal weight.
```

### 9.3 Context blob structure
```python
context = {
    "region": {"name": "...", "state": "...", "type": "..."},
    "climate": {"avg_temp_c": ..., "climate_zone": "...", ...},
    "geography": {"elevation_m": ..., "terrain": "..."},
    "athlete_summary": {
        "olympic_total": 47,
        "paralympic_total": 12,
        "top_olympic_sports": [("Swimming", 18), ("Track", 9), ...],
        "top_paralympic_sports": [("Para Swimming", 5), ("Para Track", 3), ...],
        "era_distribution": {"pre-1950": 4, "1950-2000": 22, "2000+": 33}
    },
    "infrastructure": {
        "pools": 12,
        "tracks": 8,
        "adaptive_facilities": 3,  # surface this prominently
        ...
    },
    "school_programs": {...}
}
```

### 9.4 Discovery Mode prompt (high-value)
```
Given this list of US regions ranked by athletes-per-capita per sport,
identify 5 SURPRISING hubs — places that have produced disproportionately
high numbers of athletes in specific sports relative to their size or
expected profile.

For each surprise, explain in 2 sentences what could be unusual about it,
using conditional phrasing.

Treat Olympic and Paralympic surprises with equal weight — surface at least
2 Paralympic-related discoveries.
```

### 9.5 Cost discipline
- Cache all narratives at the (region_id, prompt_version) key for 24h.
- Cap conversation agent context at 8k tokens.
- Use Flash for any call where reasoning depth is not the bottleneck.
- Set up Cloud Billing budget alerts at $25, $50, $75.

## 10. Frontend UX Principles

### 10.1 Information hierarchy on a region page
1. Region name + map pin (top)
2. Headline counts: Olympic | Paralympic | Total — equal visual weight
3. Sport breakdown — parallel charts for Olympic and Paralympic
4. Why-Engine narrative (Gemini-generated)
5. Similar hubs (vector search results)
6. Conversational agent ("Ask anything about this region")

### 10.2 Parity check for every screen
Before merging any UI change, verify:
- Olympic and Paralympic counts have identical visual treatment (same fonts, same colors, same chart sizes)
- Para sports are not behind a tab/toggle
- Adaptive infrastructure is surfaced as prominently as standard infrastructure

### 10.3 Conditional phrasing in static copy
All static UI copy uses conditional phrasing too — not just Gemini outputs. Example: "Hubs that could help nurture future Team USA athletes" not "Hubs that produce Team USA athletes."

## 15. Working with Claude Code on This Project

When asked to make changes:
1. **Always check the compliance section (11) first.** A feature that violates compliance is not a feature, it's a disqualification risk.
2. **Default to aggregate outputs.** If a code path could surface individual athlete data to the UI, refactor.
3. **Conditional phrasing is a hard requirement.** Any string output, whether static or Gemini-generated, that implies geography causes athletic outcomes is a bug.
4. **Olympic/Paralympic parity is structural.** Any UI component that displays one must display the other with equal weight by default.
5. **Cost discipline matters.** $100 in credits has to last through judging (May 12 - June 10). Cache Gemini calls. Don't redeploy on every push if not needed.
6. **The 3-minute video is the deliverable.** Build with the demo storyboard (section 12) as the north star, not feature completeness.

When in doubt, the rules document at the project root (`TeamUSA-GoogleCloud.md`) is the source of truth.
